# Story 1.6: Maquina de Estados da Conversa (FSM)

## Objetivo

Implementar a maquina de estados finitos (FSM) que controla o fluxo da conversa entre Pedro e o Koda. A FSM decide em qual "modo" o Koda esta (onboarding, aula, portao, quiz, duvida, pausa) e quais transicoes sao validas. Sem ela, o Koda nao sabe se deve dar aula, avaliar um portao ou responder uma duvida.

## Acceptance Criteria

- [x] AC1: `src/core/state-machine.ts` exporta funcao `transition(currentState: ConversationStateEnum, intent: IntentType, context: StateContext): TransitionResult`
- [x] AC2: Tipo `ConversationStateEnum` inclui os 12 estados: `IDLE`, `ONBOARDING`, `HUB`, `LESSON`, `GATE_1`, `GATE_2`, `GATE_3`, `QUIZ`, `DOUBT`, `BREAK`, `REVIEW`, `PLAYGROUND`
- [x] AC3: Interface `StateContext` definida com: `isNewUser: boolean`, `onboardingCompleted: boolean`, `currentModuleId: number | null`, `currentConceptId: string | null`, `gateAttempts: number`, `contextStack: string[]`
- [x] AC4: Interface `TransitionResult` definida com: `newState: ConversationStateEnum`, `action: string` (descricao da acao a tomar), `contextUpdate: Partial<StateContext>`
- [x] AC5: Transicao `IDLE -> ONBOARDING` quando usuario e novo (`isNewUser: true` e `onboardingCompleted: false`)
- [x] AC6: Transicao `IDLE -> HUB` quando usuario existente envia saudacao
- [x] AC7: Transicoes de onboarding: `ONBOARDING` progride internamente (step 1->2->3->4) via context. Apos step 4, transiciona para `HUB`
- [x] AC8: Transicoes do HUB: `HUB -> LESSON` (intent `lesson_continue` ou mood "focado"), `HUB -> QUIZ` (intent `quiz_answer`), `HUB -> DOUBT` (intent `doubt`), `HUB -> BREAK` (intent `mood_check` com "cansei"/"cansado")
- [x] AC9: Transicoes de aula: `LESSON -> GATE_1` (explicacao concluida), `GATE_1 -> GATE_2` (portao 1 aprovado), `GATE_1 -> LESSON` (portao 1 reprovado — re-explicar), `GATE_2 -> GATE_3` (portao 2 aprovado), `GATE_2 -> GATE_2` (tentativa errada, max 3), `GATE_3 -> HUB` (portao 3 aprovado)
- [x] AC10: Transicao de duvida com pilha: `DOUBT` empilha estado anterior no `contextStack`. Quando duvida e respondida, desempilha e volta ao estado anterior
- [x] AC11: `src/db/queries/conversation-state.ts` exporta funcoes: `getState(userId: string): Promise<ConversationState | null>`, `setState(userId: string, state: ConversationStateEnum, context: object): Promise<void>`
- [x] AC12: Estado persistido na tabela `conversation_state` do Supabase (uma linha por usuario, upsert)
- [x] AC13: Se transicao invalida (ex: `BREAK -> GATE_2`), a FSM retorna o estado atual sem mudanca e loga warning
- [x] AC14: Tabela de transicoes documentada como constante exportada em `src/core/state-machine.ts` para facil visualizacao e manutencao
- [x] AC15: Testes unitarios cobrindo pelo menos 10 transicoes: novo usuario, usuario existente, fluxo completo LESSON->GATE_1->GATE_2->GATE_3->HUB, duvida com empilhamento, transicao invalida

## Technical Notes

- A FSM e a Etapa 5 do pipeline (Story 1.4)
- O estado e armazenado na tabela `conversation_state` com coluna `context` (JSONB) para dados extras (module_id, concept_id, gate, attempts, stack)
- Referencia de estados e transicoes: `docs/architecture/fullstack-architecture.md` secao 6
- O `contextStack` e essencial para o modo DOUBT — permite voltar ao estado anterior apos responder duvida
- A FSM deve ser pura (sem side effects) — persistencia e feita separadamente
- Usar pattern de tabela de transicoes (lookup table) em vez de cadeia de if/else
