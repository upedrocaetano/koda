# Story 1.2: Schema Supabase & Migrations

## Objetivo

Aplicar a migration SQL que cria todas as tabelas, enums, indices e politicas RLS no Supabase. Sem o banco de dados, o Koda nao consegue armazenar usuarios, progresso, sessoes ou gamificacao. Pedro precisa que seus dados sejam persistidos desde a primeira interacao.

## Acceptance Criteria

- [x] AC1: Migration `supabase/migrations/00001_initial_schema.sql` aplicada com sucesso no Supabase (via `supabase db push` ou dashboard) — SQL pronto, aplicação depende de credenciais
- [x] AC2: 6 tipos enumerados criados: `user_level`, `gate_type`, `interaction_type`, `conversation_state_enum`, `mood_type`, `gamification_type`
- [x] AC3: 10 tabelas criadas: `users`, `modules`, `concepts`, `progress`, `sessions`, `interactions`, `gamification`, `conversation_state`, `reengagement_log`, `difficulty_signals`
- [x] AC4: Funcao `trigger_set_updated_at` criada e triggers ativos em `users`, `progress`, `conversation_state`
- [x] AC5: RLS habilitado em todas as 10 tabelas
- [x] AC6: Politicas RLS criadas: `users` (select/update own, insert service), `modules`/`concepts` (select public), `progress`/`sessions`/`conversation_state` (select/insert/update own), `interactions`/`gamification`/`reengagement_log` (select own, insert service), `difficulty_signals` (select/insert/update own)
- [x] AC7: Indices criados conforme schema: `idx_users_last_active_at`, `idx_users_dropout_risk`, `idx_modules_phase_order`, `idx_concepts_module_id`, `idx_progress_user_id`, `idx_sessions_user_id`, `idx_sessions_user_started`, `idx_interactions_session_id`, `idx_interactions_user_created`, `idx_gamification_user_id`, `idx_reengagement_user_day`

- [x] AC9: Constraint UNIQUE funcional em: `users.phone`, `progress(user_id, concept_id)`, `concepts(module_id, order_index)`, `gamification(user_id, type, name)`, `conversation_state.user_id`, `difficulty_signals(user_id, concept_id)`

## Technical Notes

- Migration ja existe em `/Users/pedrocaetano/Koda/supabase/migrations/00001_initial_schema.sql` — 517 linhas, completa
- O schema usa `gen_random_uuid()` (extensao `pgcrypto`) para PKs
- Backend Hono usa `SUPABASE_SERVICE_ROLE_KEY` que bypassa RLS automaticamente
- Web App usara `SUPABASE_ANON_KEY` que respeita as politicas RLS
- Tipos TypeScript em `src/db/schema.ts` devem usar os mesmos nomes das colunas do SQL
- Referencia completa do schema: `docs/architecture/database-schema.md`
