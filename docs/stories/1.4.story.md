# Story 1.4: Pipeline de Mensagem (10 Etapas)

## Objetivo

Implementar o pipeline completo de processamento de mensagem em 10 etapas — do recebimento da mensagem no webhook ate o log final no banco. Este e o coracao do Koda: cada mensagem do Pedro passa por esse pipeline antes de gerar uma resposta. Cada etapa e uma funcao isolada, testavel e com responsabilidade unica.

## Acceptance Criteria

- [x] AC1: `src/core/pipeline.ts` exporta funcao `processMessage(input: PipelineInput): Promise<PipelineResult>` que orquestra as 10 etapas sequencialmente
- [x] AC2: Interface `PipelineInput` definida com: `phone`, `text`, `messageType` ("text" | "audio" | "image"), `messageId`, `rawPayload`
- [x] AC3: Interface `PipelineResult` definida com: `success`, `userId`, `responseText`, `intent`, `xpEarned`, `newState`, `latencyMs`, `error?`
- [x] AC4: **Etapa 1 — Receive & Parse** (`src/core/pipeline/receive.ts`): recebe input bruto, normaliza texto (trim, limit 2000 chars), detecta tipo de midia. Se audio, marca para transcricao futura (retorna texto placeholder por enquanto)
- [x] AC5: **Etapa 2 — Identify User** (`src/core/pipeline/identify-user.ts`): busca usuario por phone no Supabase. Se nao existe, cria registro com `phone` e defaults. Retorna `User` com `isNew: boolean`
- [x] AC6: **Etapa 3 — Rate Limit** (`src/core/pipeline/rate-limit.ts`): verifica se usuario enviou < 10 msgs no ultimo minuto (in-memory counter). Se excede, retorna mensagem "Calma, estou processando sua mensagem anterior..." e interrompe pipeline
- [x] AC7: **Etapa 4 — Classify Intent** (`src/core/pipeline/classify-intent.ts`): chama o classificador de intencao (Story 1.5) passando texto e estado atual. Retorna `{ intent: string, confidence: number }`
- [x] AC8: **Etapa 5 — State Machine** (`src/core/pipeline/state-transition.ts`): consulta estado atual e decide transicao com base no intent (Story 1.6). Retorna `{ previousState, newState, action }`
- [x] AC9: **Etapa 6 — Build Context** (`src/core/pipeline/build-context.ts`): monta prompt em ate 6 camadas (base personality + mode + profile + curriculum + history + output instructions). Retorna `{ systemPrompt: string, userMessage: string }`
- [x] AC10: **Etapa 7 — AI Engine** (`src/core/pipeline/ai-engine.ts`): chama Claude Sonnet com o contexto montado. Timeout de 15s. Retorna structured output: `{ response_text, decisions: { gate_passed, xp_earned, next_state, mastery_update } }`
- [x] AC11: **Etapa 8 — Post-Process** (`src/core/pipeline/post-process.ts`): extrai decisoes do JSON do Claude, calcula XP, atualiza progresso e gamificacao no Supabase
- [x] AC12: **Etapa 9 — Format & Send** (`src/core/pipeline/format-send.ts`): formata resposta para WhatsApp (bold com *, code com ```, emojis), quebra em msgs < 2000 chars, envia via Evolution API
- [x] AC13: **Etapa 10 — Log & Update** (`src/core/pipeline/log-update.ts`): salva interacao na tabela `interactions`, atualiza sessao na tabela `sessions`, atualiza `last_active_at` do usuario
- [x] AC14: Pipeline mede latencia total (start-to-finish) e loga: `Pipeline completed in {X}ms for user {phone}`
- [x] AC15: Se qualquer etapa falha, pipeline loga o erro e envia mensagem de fallback para o usuario: "Desculpa, tive um problema. Tenta de novo em alguns segundos?"
- [x] AC16: Webhook em `src/routes/webhook.ts` (Story 1.3) agora chama `processMessage()` apos parsing

## Technical Notes

- O pipeline e SEQUENCIAL — cada etapa depende da anterior
- Etapas 4, 5, 6, 7 dependem de Stories 1.5 e 1.6. Implementar com interfaces/stubs que serao preenchidos quando essas stories forem feitas
- Etapa 6 (Context Builder): no MVP, montar pelo menos Layer 1 (personalidade base de `prompts/base-personality.md`) e Layer 6 (output instructions). Layers 2-5 serao expandidas em stories posteriores
- Etapa 7 (AI Engine): usar `@anthropic-ai/sdk` com `claude-sonnet-4-20250514`. Pedir structured output via system prompt com JSON format
- Etapa 9: WhatsApp markup: `*bold*`, `_italic_`, `\`\`\`code\`\`\``, emojis unicode
- Referencia completa do pipeline: `docs/architecture/fullstack-architecture.md` secao 4
- SLA total esperado: 3-10 segundos (< 15s)
